# AI Chat API

Асинхронный REST API для взаимодействия с LLM с поддержкой потоковой генерации ответов, хранения истории диалогов в PostgreSQL и логирования использования токенов.

Проект построен на принципах чистой архитектуры и ориентирован на расширяемость: реализация не зависит от конкретной языковой модели или поставщика AI-сервиса.

---

# Возможности

* Асинхронный чат с LLM
* Потоковая (streaming) генерация ответов
* Сохранение истории диалогов в PostgreSQL
* Автоматическое создание новых диалогов
* Логирование использования токенов
* Поддержка нескольких LLM-провайдеров через единый интерфейс
* Контейнеризация с помощью Docker Compose

# Слои приложения

## Controllers

Отвечают за обработку HTTP-запросов и формирование HTTP-ответов.

Не содержат бизнес-логики.

---

## Services

Содержат бизнес-логику приложения.

Сервис ничего не знает о FastAPI, SQLAlchemy или конкретной реализации LLM.

Взаимодействует только через абстракции.

---

## Repository

Инкапсулирует работу с PostgreSQL.

Отвечает исключительно за операции чтения и записи данных.

Не управляет транзакциями.

---

## Unit of Work

Отвечает за жизненный цикл транзакции.

Каждая бизнес-операция выполняется в собственной транзакции.

Используется совместно с `async with`, автоматически закрывает сессию и выполняет rollback при возникновении исключений.

---

## Clients

Содержат реализации клиентов различных LLM.

В настоящий момент доступны:

* Ollama
* OpenAI

Все клиенты реализуют единый интерфейс и могут быть заменены без изменения бизнес-логики.

---

## Models

Содержат Pydantic-модели, используемые между слоями приложения.

Не зависят от SQLAlchemy.

---

## Schemas

SQLAlchemy ORM-модели базы данных.

---

# Применяемые принципы проектирования

Проект реализован с использованием современных практик разработки backend-приложений.

## SOLID

* **S** — каждый класс имеет одну ответственность.
* **O** — приложение открыто для расширения и закрыто для модификации.
* **L** — реализации клиентов и репозиториев взаимозаменяемы.
* **I** — используются небольшие специализированные интерфейсы.
* **D** — зависимости передаются извне через внедрение зависимостей.

---

## DRY

Не допускается дублирование бизнес-логики.

---

## KISS

Каждый компонент отвечает только за свою область ответственности.

---

## Dependency Injection

Все зависимости создаются в одном месте (`dependencies.py`) и внедряются через FastAPI.

Сервис не создает репозитории, клиентов или Unit of Work самостоятельно.

---

## Repository Pattern

Доступ к данным полностью инкапсулирован внутри репозитория.

Это позволяет заменить PostgreSQL на другую систему хранения без изменения бизнес-логики.

---

## Unit of Work

Управляет транзакциями приложения.

Позволяет гарантировать атомарность бизнес-операций и корректно работать с потоковой генерацией ответов.

---

## Factory Pattern

Используется для создания конкретной реализации LLM-клиента на основании конфигурации приложения.

Выбор клиента производится автоматически через переменную окружения.

---

## Protocols

Все внешние зависимости описаны через `Protocol`.

Благодаря этому бизнес-логика не зависит от конкретных реализаций клиентов или репозиториев.

# Запуск проекта

## 1. Клонировать репозиторий

```bash
git clone <repository_url>

cd AI-Chat-API
```

---

## 2. Создать файл `.env`

Заполнить переменные окружения согласно примеру выше.

---

## 3. Запустить контейнеры

```bash
docker compose up --build
```

---

## 4. Модели Ollama

После запуска контейнеров модель загружается самостоятельно благодаря

```bash
entrypoint: >
      sh -c "
        sleep 10 &&
        ollama pull ${LLM_MODEL}
      "
```

После завершения загрузки модель сохраняется в Docker Volume и повторная загрузка не потребуется.

---

## 5. Проверить список моделей

```bash
docker exec -it ai-chat-ollama ollama list
```

---

## 6. Открыть приложение

API:

```
http://localhost:8000
```

Swagger UI:

```
http://localhost:8000/docs
```

Adminer:

```
http://localhost:8080
```

# Особенности реализации

* Полностью асинхронный стек (`asyncio`, `AsyncSession`, `asyncpg`).
* Потоковая передача ответов без ожидания завершения генерации.
* Сохранение истории диалогов в PostgreSQL.
* Логирование использования токенов после завершения генерации.
* Независимость бизнес-логики от конкретного AI-провайдера.
* Управление транзакциями через Unit of Work.
* Расширяемая архитектура за счет использования Protocol, Factory и Dependency Injection.
